# 运行流程

本页说明 Revornix 当前产品从用户请求到后台处理、再到公开页面与通知触达的主要运行链路，帮助你快速理解系统在运行时如何协同工作。

## 总览

Revornix 当前运行时主要由以下部分组成：

- `web`：用户可见的主产品前端
- `api`：同步业务 API 与主编排入口
- `celery-worker`：异步工作流与重任务执行层
- `hot-news`：独立热搜聚合服务
- PostgreSQL / Redis / Milvus / Neo4j / MinIO：底层数据与文件基础设施

其中，`web` 负责交互与展示，`api` 负责状态变更与访问控制，`celery-worker` 负责长耗时处理，`hot-news` 负责热搜聚合。

## 1. 用户访问主链路

大多数产品页面都遵循同一条主链路：

1. 浏览器访问 `web`
2. `web` 根据页面动作调用 `api`
3. `api` 查询或写入 PostgreSQL、Redis、Milvus、Neo4j、文件系统等后端资源
4. `api` 返回同步结果
5. `web` 刷新页面状态并展示结果

这条链路覆盖了登录、设置、文档列表、专栏详情、控制面板、Revornix AI 设置、通知管理等大多数同步交互。

## 2. 文档处理链路

文档相关能力通常包含“同步落库 + 异步加工”两段：

<ProcessDiagram
	title='文档处理流程'
	steps={[
		{ title: '提交文档', description: '支持速记、链接、文件和音频入口。' },
		{ title: 'API 创建记录与任务', description: '完成校验、落库和原始文件写入。' },
		{ title: 'Worker 执行异步加工', description: '解析、转写、摘要、向量化、图谱等流程继续推进。' },
		{ title: '结果回写', description: '处理产物回写数据库和文件系统。' },
		{ title: '前端展示结果', description: '详情页展示阶段状态、重试入口和最终结果。' },
	]}
/>

1. 用户在 `web` 中提交链接、文件、音频或快速笔记
2. `api` 完成权限校验、默认资源校验、文档记录创建和原始文件写入
3. `api` 创建文档处理任务，并把重任务交给 `celery-worker`
4. `celery-worker` 根据文档类型执行网站解析、文件解析、转写、Markdown 转换、摘要、向量化、图谱等流程
5. 处理结果回写数据库与文件系统
6. `web` 通过轮询或重新请求 API 展示最新状态

完成后，文档会逐步具备以下能力：

- Markdown 阅读
- AI 问答
- 向量检索
- 图谱生成
- 播客生成
- 被纳入专栏或今日专栏

## 3. 专栏处理链路

专栏流程同样分为同步配置与异步生成两部分：

		{ title: 'Worker 生成内容', description: '输出 Markdown、插图、播客、图谱和 PPT。' },
		{ title: '专栏详情页刷新', description: '页面展示正文、媒体区、评论和订阅状态。' },
	]}
/>

1. 用户创建或更新专栏配置
2. `api` 保存专栏基础信息、标签、协作关系、处理触发方式与调度信息
3. 当事件触发或定时触发到达时，`api` 或调度器发起专栏处理任务
4. `celery-worker` 汇总专栏文档、生成 Markdown、补充插图、生成播客、构建专栏图谱，并在需要时生成 PPT
5. 结果回写后，`web` 在专栏详情页中展示正文、媒体区、图谱、评论和订阅状态

今日专栏属于系统自动维护的专栏类型，但底层仍沿用相同的专栏处理链路，只是在创建时自动写入默认调度与默认开关。

## 4. Revornix AI 链路

Revornix AI 的运行依赖模型、引擎以及知识底座的共同配合：

1. 用户在前端输入问题或打开相关文档/专栏上下文
2. `web` 调用 `api` 的 AI 路由
3. `api` 根据当前默认模型、默认引擎、套餐能力和上下文资源完成访问控制与请求编排
4. 当请求涉及文档或专栏检索时，`api` 会联动向量索引、图谱、文件内容和 MCP 能力
5. 结果返回后由 `web` 渲染为对话、引用、Markdown 或结构化内容

这条链路同时覆盖：

- Revornix AI 对话
- 文档 AI 问答
- 专栏 AI 问答
- 基于检索与图谱扩展的回答

## 5. 通知与订阅链路

通知并不是独立的内容生成流程，而是建立在事件和定时任务之上的触达层：

1. 用户在设置中创建通知源、通知目标和通知任务
2. `api` 保存通知规则，并在需要时把调度任务注册到调度器
3. 当事件发生或定时时间到达后，通知任务被触发
4. `api` 或 `celery-worker` 生成通知内容并投递到邮件、飞书、钉钉、Telegram 或 Apple 设备

通知任务常与这些内容链路联动：

- 今日专栏总结
- 专栏更新
- 专栏订阅
- 专栏评论

## 6. 社区与公开页面链路

公开社区页面与站内私有工作区共用同一套核心数据，但走的是独立的公开访问入口：

1. 用户或搜索引擎访问 `/community`、`/section/[publish_uuid]`、`/user/[id]`、`/document/[id]`
2. `web` 的公开 SEO 页面请求公开接口
3. `api` 仅返回允许公开访问的资料、专栏、文档和用户信息
4. 页面输出公开内容、结构化数据和导航关系

这条链路让公开专栏、公开文档和公开用户主页可以直接被站外访问，同时保持私有内容隔离。

## 7. 热搜链路

热搜能力走独立服务，不经过主 API：

1. `web` 调用 `NEXT_PUBLIC_HOT_NEWS_API_PREFIX` 指向的 `hot-news`
2. `hot-news` 聚合不同来源的榜单数据
3. 统一格式结果返回给控制面板和热搜页面

因此，热搜服务在仓库中与主产品同仓，但运行上仍是一个独立进程。

## 8. 本地运行时顺序

当前本地开发最常见的运行顺序是：

1. 先通过 `docker-compose-local.yaml` 启动 PostgreSQL、Redis、Milvus、Neo4j、MinIO 等基础设施
2. 在 `api` 中执行初始化脚本
3. 手动启动 `api`
4. 手动启动 `celery-worker`
5. 手动启动 `hot-news`
6. 手动启动 `web`
7. 需要预览文档站时，再单独启动 `docs`

<Callout>
当前仓库的实际运行模型是“基础设施通过 Compose 启动，应用服务分别手动启动”，而不是所有应用服务统一由单个 Compose 文件直接拉起。
</Callout>
