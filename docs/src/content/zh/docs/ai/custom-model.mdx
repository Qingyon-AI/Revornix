# 自定义模型

Revornix 的“模型”不是单一聊天入口配置，而是会被不同工作流分别消费。  
现在最重要的两个默认模型位是：

- `Revornix AI 默认模型`
- `文档阅读 / 总结默认模型`

这两个模型的职责已经明确分开。

## 1. 模型入口的作用

### Revornix AI 默认模型

这个模型主要用于 Revornix AI 对话本身。  
当你在产品里和 Revornix AI 交互时，默认会优先走这里配置的模型。

### 文档阅读 / 总结默认模型

这个模型不仅用于文档总结。  
在这些工作流里，它还会参与：

- 文档总结
- 标题与描述生成
- 专栏内容处理
- 知识图谱相关抽取
- 某些引擎链路中的辅助文本理解

<Callout>
	所以“文档阅读 / 总结模型”现在更接近一个通用内容理解模型，而不是只负责生成一段 Summary。
</Callout>

## 2. 支持的模型协议

自定义模型以兼容 OpenAI 协议为主。  
也就是说，第三方模型只要能够通过兼容 OpenAI API 的方式接入，就可以在 Revornix 中作为可选模型使用。

## 3. 模型配置流程

模型配置大致分成三层：

1. 创建模型供应商
2. 在供应商下增加模型
3. 把某个模型设为默认用途

其中：

- 供应商名称和描述主要用于你自己区分来源
- 模型名称必须填写真实模型名，不能随意自定义
- 模型设置完成后，还需要到“默认模型选择”里把它绑定到具体用途

## 4. 模型相关配置项释义

当前用户真正会直接填写的模型配置，主要分成两层：

### 模型供应商层

也就是你在“新增模型供应商”时填写的字段。

- `name`：供应商名称，只用于你自己识别这组模型来源，例如 `OpenAI Official`、`OpenRouter`、`My Gateway`。
- `description`：供应商描述，用来补充说明用途、路由策略或备注，不直接参与模型调用。
- `api_key`：调用该供应商接口时使用的密钥。如果你的上游需要鉴权，这项通常必填。
- `base_url`：供应商接口根地址。Revornix 当前按 OpenAI Compatible 方式调用模型，所以这里通常填兼容 OpenAI API 的根路径。
- `is_public`：是否把这个供应商公开到社区。公开后别人可以看到并 Fork，但不会直接拿到你的密钥内容。

这几个字段里，真正影响调用的是：

- `api_key`
- `base_url`

### 模型层

也就是你在某个供应商下面新增“模型”时填写的字段。

- `name`：真实模型名，也就是请求里传给上游的 `model` 值，例如 `gpt-5.4`、`gpt-4o-mini`、`claude-3-7-sonnet`、`qwen-max`。
- `description`：模型说明，仅用于你自己区分用途，比如“适合聊天”“适合长文总结”“便宜测试模型”。
- `required_plan_level`：访问该模型所需的套餐等级。主要用于资源可见性和默认选择时的权限校验。
- `is_official_hosted`：是否是官方托管模型。这个字段更多用于平台运营和计费，不影响你理解上游接口参数。
- `compute_point_multiplier`：积分倍率。用于平台侧按调用量折算消耗，不改变模型本身推理行为。

这几个字段里，真正影响上游请求的是：

- `name`

<Callout>
	可以把它理解成：供应商层负责“请求发到哪、用什么密钥发”；模型层负责“这次到底调用哪个模型名”。
</Callout>

## 5. 当前模型配置项分别是什么意思

如果你现在只是在接一个 OpenAI Compatible 模型源，最常遇到的就是这三项：

- `api_key`：鉴权密钥。没有它，上游通常会直接拒绝请求。
- `base_url`：接口根地址。它决定请求最终发往哪个供应商、代理或网关。
- `model_name` / 模型 `name`：本次请求真正使用的模型标识。它决定调用的是具体哪一个模型能力。

更直白地说：

- `api_key` 决定“你有没有权限用”
- `base_url` 决定“请求发到哪里”
- `model_name` 决定“实际跑哪个模型”

## 6. 套餐与可访问性规则

现在模型不只是“能否看见”的问题，还带有访问等级限制。

模型资源支持以下访问控制：

- 模型可以设置 `required_plan_level`
- 默认模型在写入用户设置前会做一次访问校验
- 如果账号套餐不满足，模型可以出现在列表里，但会被禁用或提示不可用

这意味着：

- 你可以配置模型社区中的公开模型
- 但默认模型选择和实际使用仍会受当前套餐限制影响

## 7. 官方托管模型

模型支持“官方托管模型”这一类标记。

它的几个重要含义是：

- 模型可被标记为 `is_official_hosted`
- 官方托管模型可以配置积分倍率 `compute_point_multiplier`
- 在官方部署环境中，系统会 seed 一套 Revornix 官方模型供应商

官方初始化会创建以下模型供应商：

- `Revornix`

官方初始化会创建以下默认模型项：

- `gpt-5.4`

这里列出的是官方初始化中提供的默认模型项。  
具体请求路由会根据平台配置和资源策略决定。

## 8. 模型社区与 Fork

模型社区能力仍然存在：

- 你可以把自己的模型供应商设为公开
- 其他用户可以在社区里看到公开模型供应商
- 如果想真正使用对方的模型供应商，需要先 Fork

Fork 的意义是把这份资源纳入你自己的可用资源集合，否则默认模型选择器不会把它作为你自己的候选资源。

## 9. 和引擎的边界

当前产品里，“模型”和“引擎”不要混为一谈：

- 模型：更偏向 LLM / 推理 / 阅读 / 对话能力
- 引擎：更偏向网站解析、文件解析、播客合成、音频转写、图片生成这类专门能力

你在大多数功能里，实际需要同时配置这两类默认资源，而不是只配模型或只配引擎。
